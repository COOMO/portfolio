Portfolio / Engineering case study 作品集 / 工程案例分析 ▮ Replacing a LabVIEW station 取代 LabVIEW 工作站

Bench-test instrumentation,
rewritten in Rust. 儀器測試站，
用 Rust 重寫。

benchpress is a Windows desktop app that controls bench-test instruments over GPIB / Ethernet, lets engineers compose test sequences from drag-and-drop instrument command blocks, and produces per-run measurement reports. It replaces an existing LabVIEW station whose UI was tightly coupled to instrument logic, whose test flows were hard-coded for-loops, and whose configs lived nowhere a version-control tool could see. benchpress 是一個 Windows 桌面應用程式，透過 GPIB / Ethernet 控制儀器， 讓工程師把測試流程拆成可拖拉的指令積木，每次執行產出量測報告。 它取代的是一座既有的 LabVIEW 工作站，UI 跟儀器邏輯緊耦合、測試流程是寫死的 for-loop、 設定檔放在版本控制看不到的地方。

§ 01 · Brief § 01 · 動機

Why rewrite a working LabVIEW station? 為什麼要重寫一個能跑的 LabVIEW？

The old program ran. Engineers tolerated it. But every new instrument was a multi-day wiring exercise, every new test was a new VI, and the configuration that drove a run only existed on the bench PC. Three specific pains motivated the rewrite — and each one mapped to a concrete architectural decision. 舊程式可以跑，工程師也能忍受。但每加一台儀器都是好幾天的接線工程，每個新測項都是一個新 VI， 跑一次測試所依賴的設定檔只存在那台工作站的 PC 裡。有三個具體的痛點推動了這次重寫， 每一個都對應到一條明確的架構決策。

Pain · 01 痛點 · 01
UI coupled to instrument logic. UI 跟儀器邏輯緊耦合。
─── decoupled by ──▶ ─── 解耦方式 ──▶
Previously: supporting a new instrument meant writing a Rust device driver and integrating it into the editor — every new device required a full rebuild.

Now: add a row to instruments.toml and drop a <name>_templates.toml; the app picks it up at startup. Adding a device is a runtime config change, not a rebuild. 以往：要支援一台新儀器必須建立 device driver、整合進 sequence editor， 每加一台都要重新 build 整個程式。

現在：在 instruments.toml 加一列、放一份 <name>_templates.toml，app 啟動時動態載入。 新增儀器是改設定，不是重 build。
Pain · 02 痛點 · 02
Building test flows wasn't intuitive. 編寫測試流程不直覺。
─── rebuilt as ──▶ ─── 重做為 ──▶
Previously: existing tools had high operational friction — every change felt like writing code, not assembling a test.

Now: intuitive drag, click, and edit — no coding required. 以往：類似工具操作門檻高，每個改動都像寫程式而不是組裝測項。

現在：直覺的拖曳、點選、編輯，不寫程式也可以完成。
Pain · 03 痛點 · 03
Special-spec support was code-bound. 特殊規格的支援綁在程式碼裡。
─── flattened to ──▶ ─── 攤成 ──▶
Everything is managed as plain-text TOML — instruments, sequences, action templates, per-run sessions, all with versioned schemas under one folder. LLM-friendly — an LLM can draft a new device file from a vendor datasheet directly. Git-friendly — every change diffs like any other source. Adding a quirky vendor spec is a TOML edit, not a rebuild. 一切都用純文字 TOML 管理 — 儀器、sequence、Action 模板、 每次執行的 session 全部含版號 schema 放在同一個資料夾。 LLM 友善 — LLM 可以直接根據 vendor datasheet 草擬一份儀器檔。 Git 友善 — 每次改動跟其他原始碼一樣 diff。 新增 vendor 特殊規格是改 TOML，不是重 build。
§ 02 · Product tour § 02 · 介面導覽

Three views. One workflow. 三個畫面，一條工作流。

The product is a single executable that resolves into three workspaces — each designed for a different moment in the engineer's day. They share state through a typed in-memory registry, persist to plain TOML, and never duplicate logic. 整套產品是一個執行檔，分成三個工作畫面，對應工程師日常裡三個不同的時刻。 它們共享一份型別嚴格的 in-memory registry、持久化成純 TOML、不重複任何邏輯。

VIEW 01 · Interactive command testing 互動式指令測試

Test Panel

The bring-up surface. Add an instrument by VISA address, probe it for *IDN?, then send raw SCPI to validate a command before promoting it to a reusable Action. This is where a new instrument earns its place in the catalogue. 新儀器上線的入口。輸入 VISA 位址加入一台儀器，按 Probe 發 *IDN? 探測， 然後直接送原始 SCPI 驗證指令，沒問題之後再把它存成可重用的 Action。 新儀器就是在這個畫面取得編目權的。

The whole panel is template-driven — there is no per-instrument Rust panel. Whatever shows up in the action tree comes from the instrument's <name>_templates.toml on disk. 整個面板是 template-driven，沒有任何一台儀器有自己的 Rust 面板。 動作樹上看到的所有內容，都來自磁碟上對應儀器的 <name>_templates.toml。

  • Saves儲存 SCPI as Action with named params 把 SCPI 存成帶具名參數的 Action
  • Backend後端 visa-rs · runtime R&S → NI fallback visa-rs · runtime 偵測 R&S → NI

VIEW 02 · Drag-and-drop test scheduling 拖拉式測試排程

Sequence Editor

A three-pane editor in the style of NI TestStand: the action palette on the left, an ordered step list in the centre, and a parameter inspector with live Real Command preview on the right. Below the editor, a scrolling event log streams every executor lifecycle event in real time. 三欄式編輯器，靈感來自 NI TestStand：左邊是動作面板、中間是步驟列表、右邊是參數檢視器 （含即時 Real Command 預覽）。編輯器下方是一條會滾動的事件 log， 把執行器每一個生命週期事件都即時印出來。

Block::Parallel branches run across tokio worker threads; instrument-level mutexes serialize same-transport access without deadlocks. Variables, ${expr} substitution, and Producer-typed conditional queries live in the same model. Block::Parallel 的各分支跑在不同的 tokio worker thread 上； 每台儀器一個 mutex 把同 transport 的存取自然序列化、不會死鎖。 變數、${expr} 替換、Producer 型別的條件查詢全在同一個模型裡。

VIEW 03 · Application state management 應用程式狀態管理

Manage

The catalogue tab. Three sub-tables — Instruments, Actions, Sequences — each backed by a TOML file under data_dir(). Every row is hand-editable outside the app and survives git diff review like any other source. 編目分頁。三個子表格，Instruments、Actions、Sequences，每個都由 data_dir() 下的一份 TOML 檔支撐。每一列都可以在 app 外面手動編輯， 跟其他原始碼一樣可以被 git diff 審視。

The on-stop SCPI per instrument lives here too — Chroma e-loads ship LOAD OFF, Keysight ships INP:STAT OFF, ITECH ships INPut:STATe OFF. The truth is per-instrument because vendors don't agree, and safety isn't a place to assume. 每台儀器自己的 on-stop SCPI 也住在這裡，Chroma e-load 是 LOAD OFF、 Keysight 是 INP:STAT OFF、ITECH 是 INPut:STATe OFF。 所有權落在儀器層級，因為廠商之間沒有共識，而安全這件事不能用猜的。

§ 03 · System § 03 · 系統架構

A workspace built around one Transport. 圍繞單一 Transport 建構的 workspace。

Four crates. One UI shell. visa-rs linked in exactly one place. The block executor runs on tokio multi-thread; the UI sits on its own thread and never blocks on a VISA call. Safety cleanup runs on Stop, on app close, and on panic. 四個 crate。一個 UI 殼。visa-rs 只 link 一個地方。 指令積木執行器跑在 tokio multi-thread 上；UI 在自己的執行緒、絕不卡在任何一個 VISA 呼叫上。 安全清理在 Stop、關閉視窗、panic 時都會跑。

§ 04 · By design § 04 · 設計重點

Four properties the product guarantees. 四個產品承諾的特性。

These aren't aspirations. Each one names a concrete failure mode a lab engineer can describe in one sentence, and the code path in the repo that prevents it. 這些不是目標，是架構強制保證的事實。每一條都對應到一個實驗室工程師能用一句話講清楚的故障模式， 以及程式碼裡那條防止它發生的路徑。

Safety · three layers 安全 · 三層防線

Outputs never auto-enable. Cleanup always fires. 輸出絕不自動 enable。Cleanup 一定會跑。

Dangerous actions (Load ON, Output ON) are explicit two-stage confirmations in the UI — never an automatic toggle. Every instrument carries its own on_stop_scpi list fired on Stop, fatal error, window close, and process panic. A per-step Halt / Continue / Retry policy decides whether a single failing step triggers the cleanup chain. 危險動作（Load ON、Output ON）都是 UI 上的兩段確認 — 沒有自動 toggle。 每台儀器自己帶 on_stop_scpi 列表，Stop 按鈕、嚴重錯誤、視窗關閉、 process panic 任一情境都會觸發。每個步驟有 Halt / Continue / Retry 政策， 決定單一失敗是否觸發整條 cleanup。

Failure mode prevented: a shipped DUT damaged by an output left enabled across a Stop, a panic, or a user closing the window mid-run. 防止的故障：出貨時燒掉 DUT — 因為 Stop、panic、或使用者跑到一半關視窗， 輸出沒被關掉。

Non-blocking I/O 非阻塞 I/O

The UI thread never waits on VISA. UI 執行緒永不等 VISA。

Every instrument call goes through spawn_blocking onto a tokio worker thread before the executor sees it as an async fn. Block::Parallel branches with independent VISA sessions truly run concurrently across worker threads. Same-transport access is serialised by a per-instrument Mutex, so two branches hitting the same scope queue cleanly without deadlocking the runtime. 每個儀器呼叫都先經過 spawn_blocking 移到 tokio worker thread， 執行器看到的才是 async fn。Block::Parallel 各分支 只要 VISA session 獨立就真的在不同 worker thread 上並行跑。 同 transport 的存取由每台儀器一把 Mutex 序列化，所以兩條分支同時打到 同一台 scope 會自然排隊，不會死鎖。

Failure mode prevented: a 30-second scope timeout freezing the whole panel. The UI keeps rendering, the Stop button keeps responding. 防止的故障：30 秒的 scope timeout 凍結整個畫面。 UI 持續渲染，Stop 按鈕持續可按。

Auditable runs 可稽核執行

Every run leaves a complete trail. 每次執行都留下完整軌跡。

Each Run with Generate Report on creates sessions/<sequence>/<UTC-timestamp>/. Inside: one execution.jsonl row per executed block (Write / Query / Wait / For / Parallel / If / Shell / Print / SetVariable), screenshots saved as attachments, and an report.html rendered from the JSONL. 每次開啟 Generate Report 的執行都建立 sessions/<sequence>/<UTC-時戳>/。 裡面：每個執行過的 block（Write / Query / Wait / For / Parallel / If / Shell / Print / SetVariable）一行 JSON 寫進 execution.jsonl， 截圖以附件保存，report.html 由 JSONL 渲染而成。

Plain JSONL — pipe it through jq or load it in pandas. The HTML report's source of truth is the JSONL, not a parallel database. No vendored artefact format. No re-import dance. 純 JSONL — 用 jq 串流或 pandas 載入都行。 HTML 報告的 source of truth 就是 JSONL，沒有另一份資料庫、沒有自家格式、不用重新匯入。

Operator-first deployment 以操作員為先的部署

One folder on the desktop. Nothing hidden. 桌面上一個資料夾。沒有隱藏目錄。

The WiX 7 MSI installs to Desktop\benchpress\. Inside that folder is the executable and every config the app reads or writes — instruments.toml, sequences/, sessions/, and the per-instrument template files. The lab engineer sees one folder, double-clicks in, and the configuration sits right next to the binary that consumes it. WiX 7 包裝的 MSI 安裝到 Desktop\benchpress\。 資料夾裡面就是執行檔跟 app 會讀寫的一切 — instruments.toml、 sequences/、sessions/、各儀器的模板檔。 實驗室工程師看到桌面上一個資料夾、雙擊進去、設定就在執行檔旁邊。

Failure mode prevented: configuration drift from state scattered across %LOCALAPPDATA%, %PROGRAMDATA%, and the install directory — three places to check, three ways to be out of sync. 防止的故障：設定漂移 — 資料散落在 %LOCALAPPDATA%、 %PROGRAMDATA%、安裝目錄三個地方，三處要檢查，三種不一致的可能。

§ 05 · Roadmap § 05 · 下一步

Where benchpress is heading next. 接下來 benchpress 要走的方向。

The current build already replaces the LabVIEW station. The next three increments take it from "one engineer's bench" to a fleet-aware, LLM-augmented station any team in the lab can drive. 現在這版已經足以替換 LabVIEW 工作站。接下來這三步，是讓它從「一個工程師的工作台」 長成「實驗室任何團隊都能操作、跨機台協作、能跟 LLM 對話」的測試平台。

Next · 01 下一步 · 01

LLM-augmented operation LLM 連動

Four touchpoints, one orchestration layer: (1) a PDF manual parser reads instrument datasheets and registers their SCPI commands into instruments.toml; (2) an MCP server exposes benchpress's instruments, sequences, and run controls so an agent can drive a test conversationally; (3) an insight pass over execution.jsonl flags anomalies in finished reports; (4) sequence drafting turns a one-line natural-language brief into an editable sequence. 四個切入點、同一層 LLM 編排： (1) 儀器手冊 PDF parser，讀 datasheet 自動把 SCPI 指令註冊進 instruments.toml； (2) MCP server，把儀器、sequence、Run 控制暴露給 agent， 讓對話就能驅動測試； (3) 報告洞察，掃 execution.jsonl 標出異常； (4) sequence 草稿，一句話自然語意生成可編輯的測試流程。

「Use the instruments on Test Bench A to run a battery charge/discharge test, 50 cycles. Output the report as HTML.」 「用 testbench A 的儀器跑電池充放電測試， 週期 50 次，報告輸出 HTML 格式。」

Next · 02 下一步 · 02

Fleet control & remote runs 機台叢集與遠端執行

Today each benchpress is bound to one PC at one bench. The plan: a lightweight broker lets one operator browse, drive, and observe benchpress instances running on other PCs across the lab — start a sequence on Bench B from Bench A's screen, watch its execution.jsonl stream live, pull its report.html when it's done. 現在每一台 benchpress 都綁在一張工作台的一台 PC 上。 計劃：架一個輕量 broker，讓單一操作員從一張螢幕瀏覽、操作、監看實驗室裡其他 PC 上的 benchpress，在 A 工作台啟動 B 工作台的 sequence、即時看著 execution.jsonl 串流、結束時直接抓 report.html。

A fleet view aggregates active runs and completed sessions across all stations into a single dashboard, with per-bench filters. No vendored daemon — runs still write to local sessions/ first, then sync. 叢集視圖把所有工作台的進行中執行與完成 session 匯總在一個儀表板，可依工作台篩選。不引入專屬 daemon， 執行先寫進本地 sessions/，再同步出去。

extends → runtime · sessions/ 延伸自 → runtime · sessions/
Next · 03 下一步 · 03

Resource-aware scheduling 資源感知排程

Today, Block::Parallel branches run in parallel only when they happen to touch independent instruments on the same bench. The scheduler can't see what's idle on the other benches in the fleet. 目前 Block::Parallel 只在恰好打到同一工作台上不同儀器的分支才真的並行。 排程器看不到叢集裡其他工作台有哪些儀器正閒著。

The plan: tag each step with the instruments it claims, let the scheduler see live availability across the fleet, and fan independent steps onto idle benches — shortening the wall-clock time of a long sequence without changing what the sequence describes. 計劃：每個步驟標出它要佔用的儀器，排程器看著整個叢集的即時可用性， 把彼此無關的步驟散到閒置的工作台上跑， 在不改 sequence 邏輯的前提下壓縮一條長測試的牆鐘時間。
